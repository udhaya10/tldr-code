## Goal

Close the three remaining noise sources after v0.3.0:

1. **Caveat B (Gap 4 from v0.2.4 ast-infrastructure deepdive)** — DTO-typed sources lose field-level decorator semantics. `@Body() body: CreateUserDto` is detected as a tainted source, but the engine never consults `CreateUserDto`'s `@IsEmail` / `@IsInt` / `@MinLength` decorators because there is no project-wide type-to-decorator registry. Pydantic field types (`EmailStr`, `conint`, `constr`) and Java Bean Validation (`@Email`, `@Min`, `@Max`) suffer the same blindness.

2. **Cross-procedural taint** — taint analysis runs per-function. Flow shaped `f1 -> g -> f2 -> sink` is invisible if `g` is defined in a different file or different translation unit. v0.3.0's SSA (M1b) and structural AST (M2) close intra-procedural classes only. There is no function-summary infrastructure today.

3. **Deferred from v0.3.0 M2 per option (c)** — the sink dispatch flip from additive (regex+AST merged) to AST-preferring (sources path symmetry) was deferred because v0.3.0's AST sink banks lack 9 TypeScript framework sinks (NextJS×4 + Fastify×3 + NestJS×2) and 6 Python os.spawn* family entries that v0.2.3 M3 ships in regex banks. Flipping in v0.3.0 would silently drop those detections; v0.3.0's test suite has zero `compute_taint_with_tree` integration tests for these patterns. v0.4.0 closes the parity gaps in AST banks first, then flips dispatch.

This doc defines the architecture; v0.4.0 milestones M1-M8 (Section 6) implement it.

## Section 1 — DtoTypeIndex cross-file type registry

### 1.1 Motivation (closes Caveat B)

v0.2.4's deepdive at lines 630-651 documents the gap in concrete terms. Consider:

```ts
// CreateUserDto.ts
export class CreateUserDto {
    @IsEmail()
    email: string;

    @IsInt()
    @Min(0)
    age: number;
}

// users.controller.ts
@Post()
create(@Body() body: CreateUserDto) {
    db.query("INSERT INTO users SET email = '" + body.email + "'");
}
```

v0.3.0's engine sees `@Body()` and tags `body` tainted. It then walks `body.email` and `body.age` as tainted member-access. But it never opens `CreateUserDto.ts`, never sees `@IsEmail` on `email`, never sees `@IsInt` on `age`. The result: every DTO field is flagged identically, regardless of whether the framework has applied format validation. This is a noise source (false positives where the validator excludes the threat) and a precision floor we cannot lift without cross-file type knowledge.

The same shape applies to:

- **Python / Pydantic** — `class CreateUserDto(BaseModel): email: EmailStr; age: conint(ge=0)`. The type *is* the validator; engine must bridge type names to validation kinds.
- **Java / Bean Validation** — `@Email String email; @Min(0) int age;` on the DTO class.
- **C# / DataAnnotations** — `[EmailAddress] string Email { get; set; }; [Range(0,120)] int Age { get; set; }`.

All four ecosystems share the pattern: the source-typed DTO carries field-level decorators that the taint engine should consult before propagating.

### 1.2 Concrete shape

```rust
pub struct DtoTypeIndex {
    /// class_name -> field_name -> ValidationKind
    types: HashMap<String, HashMap<String, ValidationKind>>,
    /// class_name -> defining file (for diagnostics + invalidation)
    type_to_file: HashMap<String, PathBuf>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ValidationKind {
    /// Format-validating: class-validator @IsEmail, Pydantic EmailStr,
    /// Bean Validation @Email, DataAnnotations [EmailAddress].
    /// Validates input shape but does NOT escape — does not sanitize XSS or
    /// injection. Engine treats this as "source still tainted, but shape known".
    FormatValidated,
    /// Range/numeric: @IsInt, @Min, @Max, conint, [Range], @Size on numeric.
    /// Numeric coercion eliminates string-injection vectors at the boundary.
    /// Engine treats this as a partial sanitizer for SQL-int / command-int sinks.
    RangeValidated,
    /// No validating decorator on the field. Default — engine treats as fully
    /// tainted, identical to v0.3.0 behavior.
    NotValidated,
}
```

Storage is per-project; the index is built once before taint analysis and reused across every per-function invocation. For incremental analysis (daemon mode), invalidation keys on `type_to_file` so a single file edit only re-walks classes defined in that file.

### 1.3 Construction

Project-wide pre-pass before any taint analysis:

```rust
pub fn build_dto_index(project_root: &Path) -> Result<DtoTypeIndex, TldrError> {
    let mut index = DtoTypeIndex::default();
    for file in walk_project(project_root, &SUPPORTED_DTO_EXTENSIONS) {
        let lang = detect_language(file)?;
        let tree = parse(file)?;
        for class_node in walk_class_declarations(&tree, lang) {
            let class_name = node_text(class_node.child_by_field_name("name"));
            for field_node in walk_class_fields(&class_node, lang) {
                let field_name = node_text(field_node.child_by_field_name("name"));
                let kind = analyze_field_decorators(&field_node, lang);
                index.types
                    .entry(class_name.clone())
                    .or_default()
                    .insert(field_name, kind);
            }
            index.type_to_file.insert(class_name, file.to_path_buf());
        }
    }
    Ok(index)
}
```

`SUPPORTED_DTO_EXTENSIONS` initially: `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.java`, `.cs`. Expansion to Kotlin / Scala / Swift is non-blocking and ships in v0.4.0 M6 (per-language ValidationKind dispatch).

### 1.4 Per-language decorator dispatch

`analyze_field_decorators` is the language-specific extension point. v0.4.0 M1 ships TypeScript + Python; v0.4.0 M5 ships Java + C#; v0.4.0 M6 sweeps remaining languages.

| Language | Decorator source | Examples mapped to ValidationKind |
| --- | --- | --- |
| TypeScript / JavaScript | class-validator | `@IsEmail` → FormatValidated; `@IsInt`, `@Min`, `@Max` → RangeValidated; `@MinLength`, `@MaxLength` → FormatValidated |
| Python | Pydantic v1 + v2 | type annotation `EmailStr` → FormatValidated; `conint(...)`, `confloat(...)` → RangeValidated; `constr(regex=...)` → FormatValidated (treat regex as opaque shape) |
| Java | Bean Validation (jakarta.validation / javax.validation) | `@Email` → FormatValidated; `@Min`, `@Max`, `@Size` (numeric) → RangeValidated; `@Pattern` → FormatValidated; `@Size` on string → FormatValidated |
| C# | DataAnnotations | `[EmailAddress]` → FormatValidated; `[Range(...)]` → RangeValidated; `[StringLength(...)]` → FormatValidated; `[RegularExpression(...)]` → FormatValidated |

Per-language dispatch keys on `Language` (existing `crate::common::Language` enum).

### 1.5 Engine integration

Extend `compute_taint_with_tree` at `taint.rs:3142`:

```rust
pub fn compute_taint_with_tree(
    cfg: &ControlFlowGraph,
    refs: &VarRefMap,
    statements: &[Statement],
    language: Language,
    tree: &Tree,
    source: &[u8],
    ssa: Option<&SsaFunction>,           // v0.3.0 M1b
    dto_index: Option<&DtoTypeIndex>,    // v0.4.0 M1 (this section)
) -> Result<TaintInfo, TldrError>
```

When the source-detection arm produces a tainted variable whose declared type matches a class name in `dto_index.types`, every subsequent member-access on that variable consults the index:

- field has `ValidationKind::FormatValidated` and the sink is shape-sensitive (XSS / SQL-string / command-string) → still report flow but tag with `partial_validation: format`.
- field has `ValidationKind::RangeValidated` and the sink is numeric (SQL-int placeholder, exec-numeric) → suppress flow (numeric coercion neutralizes string-injection).
- field has `ValidationKind::NotValidated` → fully tainted (v0.3.0 behavior).
- field absent from index → fall back to v0.3.0 behavior (same as `NotValidated`).

The engine remains conservative: when in doubt, report the flow with a confidence tag, never silently suppress.

### 1.6 Caveats and non-goals

- **Generics / inheritance** — v0.4.0 M1 indexes classes by exact name match. Subclassing (`class AdminUserDto extends CreateUserDto`) is **not** unioned in v0.4.0 M1; documented as a known limitation and queued for v0.5.0.
- **Conditional validators** — class-validator's `@ValidateIf` / Pydantic's `validator(when=...)` produce conditional kinds. v0.4.0 M1 treats them as `NotValidated` to stay conservative.
- **Custom validators** — user-defined `@MyCustomValidator` decorators are unknown to the index. Engine logs a debug-level note and falls back to `NotValidated`.

This closes Caveat B without overreaching into v0.5.0 territory.

## Section 2 — Cross-procedural taint summaries

### 2.1 Motivation

`compute_taint` runs per-function. Flow shaped:

```ts
// helpers.ts
export function fetchAndStore(input: string) {
    db.query("INSERT INTO logs VALUES ('" + input + "')");
}

// route.ts
import { fetchAndStore } from './helpers';
app.post('/log', (req, res) => {
    fetchAndStore(req.body.message);  // taint flows OUT of this function
});
```

is invisible to the v0.3.0 engine. The taint stops at the call to `fetchAndStore` because the engine has no summary for what `fetchAndStore` does with its parameter.

YASA (Yet Another Static Analyzer) and similar inter-procedural analyzers solve this with **function summaries** — compact per-function descriptors of "if I'm called with tainted argument at position P, do I (a) return tainted, (b) pass it to a sink, (c) sanitize it?" — propagated bottom-up over the call graph.

### 2.2 Concrete shape

```rust
pub struct TaintSummary {
    pub function: FunctionId,
    /// Function returns tainted regardless of inputs (e.g., calls `req.body`
    /// internally). True iff at least one return-value-flow-source path exists
    /// even with no tainted parameters.
    pub baseline_taint_return: bool,
    /// Bit P set iff parameter at position P being tainted causes the return
    /// value to be tainted.
    pub tainted_returns_when: BitSet,
    /// Bit P set iff parameter at position P being tainted causes a sink call
    /// inside this function (transitively, accounting for callee summaries).
    pub passes_to_sink: BitSet,
}
```

`BitSet` is sized to the parameter arity (typically <= 8; cap at 64 for over-large signatures with a tail-truncation note).

We deliberately keep the summary **flow-insensitive over parameters** (each parameter independently tracked) for tractability. Parameter-pair correlations (`if param0 tainted AND param1 sanitized`) are not expressible. This is YASA-shape and a documented precision floor.

### 2.3 Fixpoint computation

Reuse `crate::callgraph::ProjectCallGraph` (`types.rs:1786`) as the backbone. Worklist algorithm:

```
1. Init: every function gets empty TaintSummary (all bits clear,
   baseline_taint_return = false).
2. Compute reverse topological order over the call graph (callees before
   callers; SCCs collapsed to least-fixpoint chunks).
3. For each fn in reverse topo order:
   a. Run intra-procedural taint with summaries-of-callees applied at every
      call site.
   b. Compute new TaintSummary from the result.
   c. If new summary differs from previous, mark callers dirty.
4. Re-process dirty functions until no summary changes.
5. Bound iterations to MAX_FIXPOINT_ITER = 50 to guarantee termination on
   pathological recursion depths.
```

Recursive call cycles converge by least fixpoint (start from empty, can only add bits, finite lattice). The 50-iteration bound is a defensive ceiling; in practice fixpoint reaches in O(call-graph-depth) iterations (~5-10 for typical projects per YASA empirical claims).

### 2.4 Caller-site application

When the intra-procedural pass at function `f` encounters a call to `g(arg0, arg1, ..., argN)`:

```
let summary = summaries.get(g).unwrap_or(&empty);
for (p_idx, arg) in args.enumerate() {
    if taint_set.contains(arg) {
        if summary.tainted_returns_when[p_idx] || summary.baseline_taint_return {
            taint_set.insert(call_result_var);
        }
        if summary.passes_to_sink[p_idx] {
            report_flow(arg, synthetic_sink_for(g));
        }
    }
}
if summary.baseline_taint_return && !args_tainted {
    taint_set.insert(call_result_var);  // g itself reads a source
}
```

The "synthetic sink" reported at the call site keeps the diagnostic anchored to the caller (where the user can act) while attributing through the callee summary.

### 2.5 Reuse of existing infrastructure

- `crate::callgraph::ProjectCallGraph` provides the call-graph backbone. v0.3.0 doesn't have explicit cross-procedural users, but the call graph itself is built and tested.
- Existing per-function `compute_taint_with_tree` becomes the intra-procedural primitive, called once per fixpoint round per function.
- v0.3.0 M1's SSA def-use chains feed naturally into per-function summary computation (parameter -> return propagation traces through SSA, not substring).
- v0.3.0 M2's structural AST member-access matching ensures summary correctness for member-access-heavy callees.

### 2.6 Boundaries and non-goals

- **Indirect calls** (function pointers, method dispatch via virtual tables, `eval`) — v0.4.0 M2 handles only direct calls and resolved virtual dispatch when the call graph successfully resolves the target. Unresolved callees produce `summary = empty` (conservative — no flow attributed, possible false negative). Documented.
- **Higher-order functions** — `[].map(fn)` where `fn` is tainted-flowing — out of scope for v0.4.0 M2; v0.5.0 candidate.
- **Concurrency** — fixpoint assumes single-threaded execution semantics. Cross-thread taint is out of scope.

## Section 5 — Testing strategy

### 5.1 Regression matrix on tldr-code itself

Run v0.4.0 against tldr-code's own source tree as the primary regression workload. Baseline: count flows reported on `cargo run -p tldr-cli -- secure crates/` under v0.3.0 vs v0.4.0.

Acceptance:
- **30% fewer false positives** (driven primarily by DtoTypeIndex format/range validation kinds and TaintSummary's accurate cross-file flows replacing v0.3.0's "stop at function boundary" overreporting).
- **0% false negatives relative to v0.3.0** — every v0.3.0-reported true positive must remain reported in v0.4.0. Cross-procedural finds NEW true positives v0.3.0 missed.

The matrix is materialized as a snapshot test under `crates/tldr-cli/tests/v040_regression_snapshot/` with the v0.3.0 baseline checked in. Diff = false-positive removed (acceptable) or true-positive lost (CI block).

### 5.2 Per-language fixture growth from ~3 to ~10

v0.3.0 has roughly 3 fixtures per language family (per the v0.2.4 taint language survey). v0.4.0 grows the per-language fixture surface to ~10:

- **1 baseline** — single-function intra-procedural flow (preserves v0.3.0 coverage).
- **2 cross-file** — function summary path: source in file A, sink in file B reached via call.
- **2 DTO** — validation-kind path; only for languages with class-validator equivalents (TS, JS, Python, Java, C#). Languages without DTO ecosystems get 2 extra cross-file fixtures.
- **2 negative** — assert no flow where summary correctly excludes; defends against summary regressions producing false positives.
- **3 stress** — large summaries (≥8 parameters), deep call chains (≥6 hops), recursive cycles. Defends fixpoint convergence.

Per-language: 16 lang families × ~10 fixtures = **160 cells**, fits alongside the existing exhaustive_matrix.

### 5.3 Sink-parity regression matrix (Section 7)

Add integration tests for compute_taint_with_tree on:

- TS NextJS framework: `redirect`, `dangerouslySetInnerHTML`, `NextResponse.json`, `Next.redirect` / `Response.redirect`.
- TS Fastify framework: `reply.send`, `reply.redirect`, `reply.header`.
- TS NestJS framework: `res.send` / `res.redirect` / `res.json`, `Response.send` / `Response.redirect` / `Response.json`.

Each integration test:

2. Routes the source to the sink under test.

Pre-flip dispatch (v0.3.0 baseline): regex fallback covers detection (preserves v0.3.0 behavior under additive merge).
Post-flip (v0.4.0 M7): AST exclusively (matches sources path symmetry). Both states tested under conditional compilation flag during the flip transition window.

## Section 7 — Sink dispatch flip + AST sink-parity bank work (deferred from v0.3.0 M2)

### 7.1 Why this section exists

v0.3.0 M2 chose **option (c)** per `contract.json` § `assertions[2].m2_option_chosen` and `contract.json` § `m2_dispatch_flip_deferred`:

> Option (c) — recommended — fixes member_patterns substring matching at the 3 detect_*_ast predicates (taint.rs:2975, 3048, 3123) via extract_member_access_receiver_and_field; KEEPS current additive sink dispatch (regex+AST merged). Result: zero risk of silently dropping v0.2.3's M3 framework patterns; CHANGELOG explicitly notes dispatch flip deferred to v0.4.0 alongside cross-procedural taint work that closes the parity gap structurally.

The sink-parity gap, per `m2-ground-truth.md` § Sink Coverage Parity Matrix:

- **TypeScript** AST sink bank (`TYPESCRIPT_AST_SINKS`, taint.rs:2025-2067) is missing **9 framework sinks** that v0.2.3 M3 ships in regex banks:
  - **NextJS (4)**: `redirect`, `dangerouslySetInnerHTML`, `NextResponse.json`, `(Next|Response).redirect`.
  - **Fastify (3)**: `reply.send`, `reply.redirect`, `reply.header`.
  - **NestJS (2)**: `res.send|redirect|json`, `Response.send|redirect|json` (each a tuple expansion — net 6 entries when expanded but 2 logical patterns).

### 7.4 Risk

**MED.** Justification:

- Bank extensions are mechanical with documented patterns (LOW component risk).
- Dispatch flip is 4 LOC, but every existing additive-dedup code path becomes redundant; cleanup must be inspected for correctness (MED component risk).
- Per-language sanity sweep required because dispatch was effectively "regex always wins on overlap" pre-flip; post-flip "AST wins, regex fallback" shifts which engine reports the diagnostic — diagnostic format / message text may shift in tests (MED component risk; covered by re-baseline budget).

**Mitigation:**

- Section 7 ships in v0.4.0 Wave 1 (parallel with M1+M2) so it lands early and stabilizes through the rest of v0.4.0's cycle.
- The 30 new integration tests are the principal guard; they encode the v0.2.3 M3 framework patterns as positive expectations so any future drop is loud.

### 7.5 Why deferring was the right call for v0.3.0

For posterity (and to defend against the temptation to revisit in a hotfix): v0.3.0 M2 had three options per `contract.json` § `assertions[2].m2_option_rationale`:

- **(a) fix-first-then-flip** — extend AST banks AND flip in v0.3.0. Doable but ~+50 LOC of additive bank entries that the *brand new* structural matcher must handle correctly first time. Risk asymmetry: if the structural matcher had a bug specific to NextJS-shaped tuples, the regression would be silent (no integration tests cover that path in v0.3.0).
- **(b) skip-dispatch-flip** — never flip; ship additive merge indefinitely. Safe but accumulates technical debt.
- **(c) defer-dispatch-flip** — fix the substring-matching bug (the original M2 mandate) but keep additive dispatch. Defer flip + parity to v0.4.0 where cross-procedural taint work touches the same code path and adds the integration test foundation. **Chosen.**

This document delivers on the v0.4.0 commitment that option (c) made.

## Risks & open questions

1. **Fixpoint convergence on real-world call graphs.** The 50-iteration ceiling is defensive; if production code base is ever observed to need >50, the bound is raised but a perf alarm fires. To be measured during M3 prototype.
2. **DtoTypeIndex memory footprint.** A monorepo with thousands of DTOs holds a few hundred kilobytes of HashMap; not a real concern. Watch on first daemon-mode index build.
3. **Indirect calls drop summaries.** Conservative posture (no flow attributed) keeps engine sound (no false positives from missed callees) but loses detection. Document; v0.5.0 may add type-flow analysis to recover precision.
4. **Cross-language calls** (TS calling Python via subprocess, Python calling Java via JNI) — out of scope for v0.4.0; documented as v0.6.0 candidate.
5. **Section 7 dispatch flip side effects.** The additive-dedup logic in v0.3.0 may have masked subtle bugs; flipping exposes them. Mitigation: re-baseline budget allotted, integration tests precede the flip.
